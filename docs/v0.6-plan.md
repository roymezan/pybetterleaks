# v0.6 Plan

v0.6 is the release-machine and adoption-polish release. v0.5 proved that the
tag-to-PyPI path works, but the process still has human edges: manual approval,
post-release checks, benchmark publication, and project presentation.

The theme:

> Make releases boring and make the project easier to trust at a glance.

## Goals

- Add a clear `release/*` branch workflow.
- Add version automation that defaults to patch bumps unless a release branch
  or maintainer input states otherwise.
- Add release-branch merge tagging so merging `release/vX.Y.Z` to `main`
  creates `vX.Y.Z` and lets the tag workflow publish.
- Keep PyPI trusted publishing tokenless.
- Make the manual `pypi` environment approval flow explicit and documented.
- Add post-release audit automation for PyPI, GitHub release assets, checksums,
  and install smoke checks.
- Fix or refresh GitHub Actions usage that emits Node deprecation warnings.
- Publish benchmark outputs as CI artifacts.
- Add wheel-installed benchmark smoke tests.
- Compare benchmark results against:
  - PyBetterleaks SDK calls
  - Betterleaks CLI calls
  - a subprocess-wrapper baseline
- Add GitHub project hygiene files:
  - issue templates
  - PR template
  - `SECURITY.md`
- Add a dedicated Git scanning guide.
- Add more examples to generated API docs.
- Keep the README and PyPI project page visually sharper and honest.

## Non-Goals

- No streaming scan API.
- No tracked, staged, diff, or history Git scopes.
- No provider wrappers for GitHub, GitLab, Hugging Face, S3, or remote Git URLs.
- No Linux arm64 wheel commitment unless demand or CI capacity changes.
- No musllinux/Alpine wheels.
- No sdists.
- No SBOM requirement yet.
- No artifact signing beyond current GitHub artifact attestations.
- No Betterleaks version bump unless explicitly planned and tested.

## Implementation

### Release Branch Workflow

- Treat `release/*` branches as release preparation branches.
- Run the same quality gates as `main`.
- Run wheels and E2E before a tag is created.
- Do not publish to PyPI from ordinary branch pushes.
- Preserve the `pypi` GitHub environment approval step before upload.
- Document the branch flow in `docs/release-checklist.md`.

Decision:

- `release/*` branches are preparation branches. Merging `release/vX.Y.Z` to
  `main` creates tag `vX.Y.Z` after verifying `pyproject.toml` matches the
  branch version. PyPI publishing remains tag-triggered.

### Version Automation

- Add a small script for version updates across:
  - `pyproject.toml`
  - `python/pybetterleaks/_version.py`
  - `uv.lock`
  - any release docs that mention the package version
- Default to a patch bump.
- Allow explicit minor or major/pre-release bumps through an input or command
  argument.
- Add tests that prove version files stay synchronized.

Decision:

- The version automation updates version-bearing project files only. Changelog
  edits remain human-reviewed, and GitHub generated release notes remain the
  public release-note source attached to tags.

### Post-Release Audit

- Add a workflow or script that checks a just-published version:
  - PyPI exact-version install from a temporary virtual environment
  - `pip install --only-binary=:all: pybetterleaks==<version>`
  - `betterleaks_version()`
  - synthetic `scan_text`
  - GitHub release exists
  - `SHA256SUMS` is attached
  - checksums match downloaded release artifacts
  - no musllinux wheels were published
- Keep the audit runnable locally with `uv run`.

### Benchmark Publication

- Keep current synthetic benchmarks reproducible.
- Add a subprocess-wrapper baseline for honest positioning.
- Add wheel-installed benchmark runs from clean virtual environments.
- Upload benchmark JSON/Markdown as CI artifacts.
- Keep the benchmark docs page refreshed from generated benchmark Markdown.
- Avoid broad performance claims outside measured scenarios.

### GitHub Project Polish

- Add issue templates for:
  - bug report
  - feature request
  - platform/install issue
- Add a PR template with release-risk, tests, and docs prompts.
- Add `SECURITY.md` with private vulnerability reporting guidance.
- Add `CODEOWNERS` so branch protection can require maintainer review.
- Confirm repository topics, description, homepage, and Pages settings.

### README And Docs Polish

- Keep the README punchy:
  - hero visual
  - clear install command
  - short typed-config example
  - platform promise
  - no-runtime-subprocess story
- Add a dedicated Git scanning guide for `scan_git(..., scope="worktree")`.
- Expand API docs with examples for common public functions.
- Keep repo-only planning docs excluded from public MkDocs output.

## Acceptance Criteria

- `uv run coverage run -m pytest`
- `uv run coverage report`
- `uv run ruff check .`
- `uv run mypy python`
- `uv run --group docs mkdocs build --strict`
- Docker E2E passes.
- Wheels workflow passes on supported platforms.
- Release branch workflow is documented and tested enough to use for the next
  tag.
- Version automation has unit coverage.
- Post-release audit passes against the current published package.
- Benchmark artifacts are uploaded from CI.
- README hero renders on GitHub and PyPI.

## Deferred To Later

- Native streaming handles.
- Additional Git scopes.
- Provider and remote source wrappers.
- Linux arm64 wheels.
- SBOM generation as a default release artifact.
- Artifact signing beyond current GitHub attestations.
- Source distributions.
